Introduction to FHIR

What is FHIR?

Fast Healthcare Interoperability Resources (FHIR) is a standard for exchanging healthcare information electronically.

Key Concepts We Previously Covered:

  • Resources: The building blocks of FHIR (Patient, Observation, Medication, etc.)
  • RESTful API: FHIR uses standard HTTP operations (GET, POST, PUT, DELETE)
  • JSON/XML: Data can be represented in multiple formats
  • Interoperability: Enable different healthcare systems to communicate seamlessly

FHIR Architecture Overview

Resources
‚Üí
Data Types
‚Üí
API Layer
‚Üí
Applications

Architecture Layers:

  • Foundation Layer: Core data types, identifiers, and base resources
  • Conformance Layer: Defines how systems declare their capabilities
  • Clinical Layer: Clinical resources (Patient, Observation, Condition)
  • Financial Layer: Billing and claims resources
Today's Focus: We're diving deep into the Foundation Layer - the building blocks that make everything else work!

üî§ Data Types in FHIR

What are Data Types?

Data types define the structure and format of data elements in FHIR resources. They ensure consistency and interoperability across different systems.

Categories of Data Types

Category Description Examples
Primitive Types Simple, atomic values string, integer, boolean, date, dateTime
Complex Types Structured data with multiple elements Address, HumanName, CodeableConcept, Identifier
Special Types Domain-specific structures Quantity, Range, Period, Reference

Special Data Types

Special types are domain-specific structures used for healthcare measurements, ranges, and resource references.

Quantity - Measurements with Units

Represents a measured amount with a value and unit. Commonly used for vital signs, lab results, medication dosages, and other measurements.

Quantity Examples JSON

Range - Value Ranges

Represents a range of values with a low and high boundary. Used for normal ranges, therapeutic ranges, age ranges, and other interval measurements.

Range Example - Normal Blood Pressure JSON
{ "referenceRange": [ { "low": { "value": 90, "unit": "mmHg" }, "high": { "value": 120, "unit": "mmHg" } } ] }

Reference - Linking Resources

References connect FHIR resources together. They can be relative (same server), absolute (full URL), or logical (by identifier). This is how resources form relationships in FHIR.

Reference Examples JSON
{ "subject": { "reference": "Patient/p1" } }

Relative Reference

References a resource on the same server

Same Server JSON
{ "reference": "Patient/123" }

Absolute Reference

Full URL to a resource on any server

Cross-Server JSON
{ "reference": "https://fhir.example.org/Patient/123" }

Logical Reference

References by identifier when ID is unknown

By Identifier JSON
{ "identifier": { "system": "http://hospital.org/patients", "value": "MRN-12345" } }

üé´ Identifiers in FHIR

What is an Identifier?

An identifier is a unique label assigned to a resource to identify it across different systems. Think of it as a "business identifier" rather than a technical database ID.

Why Multiple Identifiers?

A single patient might have multiple identifiers from different systems:

  • Medical Record Number (MRN)
  • Social Security Number (SSN)
  • Driver's License Number
  • Insurance Member ID
  • Passport Number

Identifier Components

System

Defines the namespace/authority that issued the identifier

Value

The actual identifier string

Use

How the identifier is used (official, temp, etc.)

Type

What kind of identifier it is

üí° Codeable Concepts

What is a CodeableConcept?

A CodeableConcept represents a value that can be coded using one or more coding systems. It allows expressing the same medical concept in multiple terminologies simultaneously.

Think of it as:

A concept that can be described using codes from different medical terminologies (LOINC, SNOMED CT, ICD-10, etc.) plus a human-readable text description.

Example: "Diabetes" can be represented in SNOMED CT, ICD-10, and your hospital's internal coding system - all at once!

CodeableConcept Structure

Element Type Description
coding Array Array of Coding objects from different code systems
text String Human-readable plain text representation

Complete CodeableConcept Example

Diabetes Diagnosis - Multiple Coding Systems

This example shows how a single diagnosis can be represented in three different coding systems:

Diabetes Coded Three Ways
{ "code": { "coding": [ { "system": "http://snomed.info/sct", "code": "73211009", "display": "Diabetes mellitus" }, { "system": "http://hl7.org/fhir/sid/icd-10", "code": "E11", "display": "Type 2 diabetes mellitus" }, { "system": "http://hospital.example.org/codes", "code": "DM-002", "display": "Diabetes Type 2" } ], "text": "Type 2 Diabetes Mellitus" } }

Why Multiple Codings?

üåê Interoperability

Different healthcare systems use different terminologies. Multiple codings ensure your data works everywhere.

üéØ Specificity

Some code systems are more detailed than others. You can provide both general and specific codes.

üåç Regional Requirements

Different countries or regions may require specific coding systems for legal or regulatory purposes.

üí∞ Billing vs Clinical

ICD codes for billing and insurance, SNOMED CT for detailed clinical documentation.

üìö Code Systems

What is a Code System?

A Code System is a collection of codes with specific meanings. It defines a set of codes and their definitions for use across healthcare applications. Think of it as a medical dictionary or catalog.

Code System Definition:

A systematically organized collection of concepts and codes with precise definitions, designed to be used consistently across different healthcare contexts.

Popular Healthcare Code Systems

SNOMED CT

Systematized Nomenclature of Medicine

Comprehensive clinical terminology covering diseases, procedures, anatomy, and more.

http://snomed.info/sct

LOINC

Logical Observation Identifiers

Standard for laboratory tests and clinical observations.

http://loinc.org

ICD-10

International Classification of Diseases

Used globally for diagnoses, billing, and health statistics.

http://hl7.org/fhir/sid/icd-10

RxNorm

Medication Nomenclature

Standardized names for medications and drug products.

http://www.nlm.nih.gov/research/umls/rxnorm

CPT

Current Procedural Terminology

Medical procedures and services for billing.

http://www.ama-assn.org/go/cpt

CVX

Vaccine Administered

Standardized codes for vaccines.

http://hl7.org/fhir/sid/cvx

FHIR-Defined Code Systems

FHIR itself defines several standard code systems that are built into the specification. These are commonly used across many resources and provide standardized values for administrative and clinical purposes.

Administrative Gender Code System

Used to represent the administrative gender of a person. This is distinct from biological sex and is used for administrative purposes like patient registration and demographics.

Administrative Gender CodeSystem JSON
{ "resourceType": "CodeSystem", "url": "http://hl7.org/fhir/administrative-gender", "name": "AdministrativeGender", "title": "Administrative Gender", "status": "active", "content": "complete", "concept": [ { "code": "male", "display": "Male" }, { "code": "female", "display": "Female" }, { "code": "other", "display": "Other" }, { "code": "unknown", "display": "Unknown" } ] }

Usage in Patient Resource

Patient.gender JSON
{ "resourceType": "Patient", "gender": "male" }

Available Codes

  • male - Male
  • female - Female
  • other - Other
  • unknown - Unknown

Observation Status Code System

Defines the registration status of an observation. This indicates whether the observation is preliminary, final, amended, or in another state. Critical for understanding the reliability and completeness of clinical data.

Observation Status CodeSystem JSON
{ "resourceType": "CodeSystem", "url": "http://hl7.org/fhir/observation-status", "name": "ObservationStatus", "title": "Observation Status", "status": "active", "content": "complete", "concept": [ { "code": "registered", "display": "Registered" }, { "code": "preliminary", "display": "Preliminary" }, { "code": "final", "display": "Final" }, { "code": "amended", "display": "Amended" }, { "code": "corrected", "display": "Corrected" }, { "code": "cancelled", "display": "Cancelled" } ] }

Usage in Observation Resource

Observation.status JSON
{ "resourceType": "Observation", "status": "final", "code": { "text": "Blood Glucose" } }

Status Codes Explained

  • registered - Observation is registered but not complete
  • preliminary - Results are preliminary and may change
  • final - Results are final and verified
  • amended - Results have been amended after finalization
  • corrected - Results were corrected
  • cancelled - Observation was cancelled

üí° Why FHIR-Defined Code Systems Matter

  • Standardization: Ensures consistent values across all FHIR implementations
  • Interoperability: Different systems can understand the same codes without custom mappings
  • Built-in Support: No need to create custom code systems for common administrative values
  • Validation: FHIR servers can automatically validate against these standard code systems

üìã Value Sets

What is a Value Set?

A Value Set is a curated collection of codes from one or more Code Systems. It defines which specific codes are valid for a particular use case or context.

Key Difference Between Code System and Value Set:

Code System: Defines all possible codes in a terminology (the entire dictionary)

Value Set: Defines which codes from one or more code systems are allowed in a specific context (selected entries from the dictionary)

Value Set Composition

SNOMED CT
+
LOINC
+
ICD-10
=
Custom Value Set

Why Use Value Sets?

üéØ Constrain Choices

Limit codes to only those relevant for a specific field or purpose.

üîó Combine Systems

Pull codes from multiple code systems into a single logical grouping.

üìÖ Versioning

Manage changes in allowed codes over time.

‚ôªÔ∏è Reusability

Share common value sets across applications.

Value Set Structure

Element Description
URL Unique canonical identifier for the value set
Compose Defines which codes to include from which systems
Include Codes to include from specified code systems
Exclude Codes to explicitly exclude (optional)
Expansion The complete list of codes after processing compose rules

Complex Value Set - Multiple Systems

Diabetes Diagnosis Value Set

A value set that includes diabetes-related codes from multiple code systems:

Multi-System ValueSet
{ "resourceType": "ValueSet", "url": "http://example.org/fhir/ValueSet/diabetes-conditions", "name": "DiabetesConditions", "title": "Diabetes Conditions", "status": "active", "description": "Codes for diabetes diagnoses", "compose": { "include": [ { "system": "http://snomed.info/sct", "concept": [ { "code": "73211009", "display": "Diabetes mellitus" }, { "code": "44054006", "display": "Type 2 diabetes mellitus" } ] }, { "system": "http://hl7.org/fhir/sid/icd-10", "concept": [ { "code": "E11", "display": "Type 2 diabetes mellitus" }, { "code": "E10", "display": "Type 1 diabetes mellitus" } ] } ] } }

Best Practices:

  • Use existing standard value sets when available
  • Create custom value sets only when needed
  • Document the purpose clearly
  • Version your value sets properly
  • Use filters for large code systems

üè• Patient Story: John's Hospital Visit

üìñ The Story Begins...

Meet John Smith, a 45-year-old man who arrives at City General Hospital complaining of frequent urination, excessive thirst, and fatigue. Let's follow his journey through the hospital system and see how FHIR represents his information at each step.

Step 1: Patient Registration

John's Personal Information

When John arrives, the registration desk collects his basic information. This is where we use Data Types to structure his data.

Complete Patient Resource
{ "resourceType": "Patient", "id": "patient-001", "identifier": [ { "use": "official", "type": { "coding": [ { "system": "http://terminology.hl7.org/CodeSystem/v2-0203", "code": "MR", "display": "Medical Record Number" } ] }, "system": "http://citygeneral.org/patients", "value": "MRN-2024-12345" } ], "name": [ { "use": "official", "family": "Smith", "given": ["John", "Michael"] } ], "gender": "male", "birthDate": "1979-03-15", "address": [ { "use": "home", "line": ["123 Oak Street", "Apt 4B"], "city": "Boston", "state": "MA", "postalCode": "02101", "country": "USA" } ], "telecom": [ { "system": "phone", "value": "+1-617-555-0123", "use": "mobile" }, { "system": "email", "value": "john.smith@email.com", "use": "home" } ] }

Doctor's Diagnosis

Dr. Sarah Johnson examines John and suspects Type 2 Diabetes. She records the diagnosis using CodeableConcept to represent the condition in multiple coding systems.

{ "resourceType": "Condition", "id": "condition-001", "subject": { "reference": "Patient/patient-001" }, "code": { "coding": [ { "system": "http://snomed.info/sct", "code": "44054006", "display": "Type 2 diabetes mellitus" }, { "system": "http://hl7.org/fhir/sid/icd-10", "code": "E11", "display": "Type 2 diabetes mellitus" } ], "text": "Type 2 Diabetes Mellitus" }, "onsetDateTime": "2024-01-15T10:30:00Z", "recordedDate": "2024-12-15T14:20:00Z" }

Blood Glucose Test

John's blood is drawn for a glucose test. The lab results are recorded as an Observation resource with coded values.

{ "resourceType": "Observation", "id": "observation-001", "status": "final", "subject": { "reference": "Patient/patient-001" }, "code": { "coding": [ { "system": "http://loinc.org", "code": "2339-0", "display": "Glucose [Mass/volume] in Blood" } ], "text": "Blood Glucose" }, "valueQuantity": { "value": 185, "unit": "mg/dL", "system": "http://unitsofmeasure.org", "code": "mg/dL" }, "effectiveDateTime": "2024-12-15T11:00:00Z", "interpretation": [ { "coding": [ { "system": "http://terminology.hl7.org/CodeSystem/v3-ObservationInterpretation", "code": "H", "display": "High" } ] } ] }

Step 4: Treatment Plan

Medication Prescription

Based on the diagnosis, Dr. Johnson prescribes Metformin. The medication is recorded using CodeableConcept with RxNorm codes.

MedicationRequest Resource
{ "resourceType": "MedicationRequest", "id": "medication-001", "status": "active", "intent": "order", "subject": { "reference": "Patient/patient-001" }, "medicationCodeableConcept": { "coding": [ { "system": "http://www.nlm.nih.gov/research/umls/rxnorm", "code": "6809", "display": "metformin hydrochloride 500 MG Oral Tablet" } ], "text": "Metformin 500mg" }, "dosageInstruction": [ { "text": "Take 1 tablet twice daily with meals", "timing": { "repeat": { "frequency": 2, "period": 1, "periodUnit": "d" } }, "doseAndRate": [ { "doseQuantity": { "value": 1, "unit": "tablet" } } ] } ] }

üìö What We Learned from John's Story:

  • Data Types structure John's personal information (name, address, dates)
  • Identifiers uniquely identify John across systems (MRN)
  • CodeableConcept represents medical concepts (diagnosis, medications) using standardized codes
  • Code Systems provide the vocabulary (SNOMED CT, ICD-10, LOINC, RxNorm)
  • Value Sets constrain which codes are valid for specific purposes

All these FHIR building blocks work together to create a complete, interoperable representation of John's healthcare journey!

üîó FHIR Resources & Relationships

What are FHIR Resources?

FHIR Resources are the basic units of exchange representing real-world healthcare concepts. Each resource has a common structure including resourceType, id, meta, and resource-specific fields.

Common Resource Structure

Every FHIR resource follows a standard structure that ensures consistency and interoperability across all healthcare systems.

Standard Resource Structure

Common Resource Elements JSON
{ "resourceType": "ResourceName", "id": "unique-id", "meta": { // Version, profile, security tags }, // Resource-specific fields... }

  • Enables generic tooling and parsers
  • Simplifies learning and implementation
  • Facilitates interoperability between systems

üßë Patient Resource

Purpose

Purpose

Represents a person (or animal) receiving healthcare services. The Patient resource stores demographic and administrative information.

Key Fields

  • identifier: Unique identifiers (MRN, SSN, etc.)
  • name: Person's name (HumanName data type)
  • gender: Administrative gender
  • birthDate: Date of birth
  • address: Contact addresses
  • telecom: Contact information (phone, email)
Patient Resource Example JSON
{ "resourceType": "Patient", "id": "p1", "name": [ { "family": "Doe", "given": ["John"] } ], "gender": "male", "birthDate": "1990-01-01" }

üí° Important Notes

  • Patient does NOT store clinical data directly - clinical information is stored in other resources (Observation, Condition, etc.) that reference the Patient
  • Patient resource represents both humans and animals - The differentiation is done using specific elements (BackboneElement) inside the Patient resource, not by using a different resource

üë®‚Äç‚öïÔ∏è Practitioner Resource

Purpose

Represents individuals who are directly or indirectly involved in the provisioning of healthcare services (doctors, nurses, therapists, etc.).

Key Fields

  • identifier: Professional identifiers (license numbers, NPI, etc.)
  • name: Practitioner's name
  • qualification: Professional qualifications and certifications
  • telecom: Contact information
  • address: Practice locations
Practitioner Resource Example JSON
{ "resourceType": "Practitioner", "id": "pr1", "name": [ { "family": "Smith" } ] }

üî¨ Observation Resource

Purpose

Stores measurements, assessments, and clinical findings. This is one of the most frequently used resources in FHIR.

Common Use Cases

  • Vital signs (blood pressure, temperature, heart rate)
  • Laboratory test results
  • Clinical findings and assessments
  • Survey responses

Key Fields

  • status: Registration status (preliminary, final, amended, etc.)
  • code: What was observed (CodeableConcept)
  • subject: Reference to Patient
  • value[x]: The actual result (valueQuantity, valueString, valueCodeableConcept, etc.)
  • effective[x]: When the observation was made (effectiveDateTime, effectivePeriod)
Observation Resource Example JSON
{ "resourceType": "Observation", "status": "final", "code": { "text": "Blood Pressure" }, "subject": { "reference": "Patient/p1" }, "valueQuantity": { "value": 120, "unit": "mmHg" } }

üí° Observation is one of the most used resources in FHIR

Almost any clinical measurement, test result, or assessment can be represented as an Observation resource, making it extremely versatile and commonly used.

üìÖ Appointment Resource

Purpose

Represents a planned healthcare event where a patient and practitioner(s) are expected to meet for a specific purpose.

Key Fields

  • status: Appointment status (proposed, booked, arrived, cancelled, etc.)
  • participant: Who is involved (Patient, Practitioner, Location, etc.)
  • start: When the appointment begins
  • end: When the appointment ends
  • serviceType: Type of service being provided
Appointment Resource Example JSON
{ "resourceType": "Appointment", "status": "booked", "participant": [ { "actor": { "reference": "Patient/p1" } }, { "actor": { "reference": "Practitioner/pr1" } } ] }

üé§ Audience Question #3

"Which resource would you use to store a patient's blood pressure: Patient or Observation? Why?"

Expected Answer:

  • Observation - because blood pressure is clinical data
  • Separation of demographics (Patient) vs clinical data (Observation)
  • Patient stores who, Observation stores what was measured

üîó References & Relationships

How Resources Connect

FHIR resources don't exist in isolation. They form a connected graph where resources reference each other to create meaningful healthcare information. FHIR uses References to link resources together. Instead of embedding full resource data, resources reference other resources by their identity.

Reference Structure

Reference Example JSON
{ "subject": { "reference": "Patient/p1" } }

Reference Formats

Relative Reference

"Patient/123"

References a resource on the same server

Absolute Reference

"https://fhir.example.org/Patient/123"

Full URL to a resource on any server

Logical Reference

{"identifier": {...}}

References by identifier when ID is unknown

üé§ Audience Question #4

"Why do you think FHIR avoids embedding full Patient data inside Observation?"

Expected Answers:

  • Avoid duplication - Patient data stored once, referenced many times
  • Single source of truth - Update Patient once, all references see the change
  • Easier updates - Change patient address in one place, not in thousands of observations
  • Reduced data size - References are much smaller than full resource copies
  • Data integrity - Prevents inconsistencies from partial updates
